WXYC · Touring Soon (#474) · prototype

Filter studies for the Touring Soon tab

Three interfaces over the same in-memory concert cache. Like triangle-shows.net, the feed is fetched once and every filter applies client-side, instantly — no spinner, no refetch. Flip the release switch to watch each interface absorb genre filters (R2) and For You (R3) one release at a time. Every control below is live.

Variant
Release

The release runway

Each stage is independently shippable and none requires rework of the previous one. The load-bearing decision is made at R1: fetch the whole window once, filter in memory — that is what makes R2 and R3 pure additions.

R1 · next release

Instant local filters

Date window, city/venue, free, all-ages — every facet already exists on the Concert schema. Fetch the full curated window from GET /concerts (today ~9 events, page limit 100), keep it in memory, filter on tap. Sold out / cancelled render as state, not filters.

Needs: nothing new — API already suffices.

R2 · following release

Genre filters

Concert lacks genre today. Backend-Service enriches at read time: headlining_artist_id → the artist's library genre (catalog/LML), falling back to triangle-shows' scraped genre field (Ticketmaster tags) for unresolved headliners. Client stays local-instant — genre is just another facet on the cached rows.

Needs: `genres: [String]` on Concert in api.yaml + BS read-path join.

R3 · after that

For You

No likes signal exists in the app yet. Stage it: (a) on-device hearts on playcut rows, matched against headlining_artist_id — “You've liked them on air,” zero backend; (b) later, semantic-index expands the match to similar artists — “Because you like Stereolab.” Privacy stays on-device until (b).

Needs: likes UI + local store; later semantic-index /similar.

Open questions

  • Rail budget vs. sheet depth. A is fastest to scan but visibly strains at three chip rows by R3. B hides everything behind one tap but absorbs R2/R3 without layout change. Does the tiny feed (~9 shows) even justify B's machinery at R1?
  • Is C a complement, not a competitor? Shelves are the natural home for For You and genre browsing, but weakest for goal-directed narrowing ("free shows in Durham this weekend"). C could be the tab's default presentation with A/B's controls layered on.
  • Smart solo/restore (tap a city = solo it, tap again = restore all — stolen from triangle-shows) is delightful on the web. Does it read as discoverable on iOS, or does it need the first-tap hint?
  • Search is deliberately out of scope here — at 9–50 events, facets beat typing. Revisit when recall climbs (BS#1604 / triangle-shows#18).
  • Filter persistence: do filters reset per visit (triangle-shows resets except hidden venues) or persist via DefaultsStorage?
